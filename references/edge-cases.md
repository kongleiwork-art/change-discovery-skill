# Edge Cases

## Scope grows during work

If a small change later needs schema, auth, public API, multi-service, or hard-to-reverse behavior:

1. Stop before more discovery or implementation.
2. Report the reclassification and updated product assessment.
3. Re-enter the product decision gate, then the model checkpoint when needed.

Do not keep coding under the old small-change assumption.

## User says “just implement”

1. Restate classification and product verdict briefly.
2. Name the main skip risk in plain language.
3. Ask for an explicit override such as **implement anyway**.
4. For medium or major work, still require plan approval or an equally explicit implementation go-ahead before editing files.

“Continue” at the model checkpoint only clears model selection, not implementation.

## User accepts reduce scope

Treat the reduced direction as the working request for research, requirements, options, and planning. Mention the abandoned larger request only as an explicit non-goal unless the user brings it back.

## User rejects reduce scope

If the user insists on the original larger request:

1. Record that insistence as an explicit assumption.
2. Reclassify if the larger scope raises risk.
3. Update the model-tier advice when needed.
4. Keep the weaker product case visible until new evidence strengthens it.

## Decline or defer, then continue anyway

If the user replies **continue anyway**:

1. Open the model checkpoint next.
2. Continue discovery after confirmation.
3. Keep the negative or deferred product recommendation visible.
4. Revise the verdict only when new project evidence or research materially strengthens the case.

## Evidence is too thin to search safely

Ask one material question before GitHub research when the missing answer would change the problem category, target user, platform, or constraint set. Do not widen the search instead of asking.

## Host cannot research GitHub

Say that research is unavailable. Ask whether to continue from project evidence and first principles. Never imply that analogues were reviewed.

## Standing pause for small changes

If the user has asked to pause even for small work, give classification, proportionate product assessment, and model advice, then stop for confirmation instead of continuing by default.
